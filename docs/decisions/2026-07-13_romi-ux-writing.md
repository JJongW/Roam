# 로미 UX Writing 카탈로그 — 행동·맥락별 발화

> 상태: **작성 중(살아있는 카탈로그)**. 목표 = "사용자의 **어떤 행동/화면**에 로미가
> **무슨 말**을 하는가"를 전부 정의. 1000+ 트리거를 지향하며 섹션별로 채워 나간다.

## 왜 이 문서 (핵심 문제)

지금 문제는 로미의 **말투(반말/존댓말)**가 아니다. **장소·화면·행동이 달라도 로미가
같은 말을 내뱉는 것** — 사용자가 보는 것/하는 것에 맞춰 말이 변하지 않는다. 로미가
"동행자"라면 **지금 이 화면·이 행동·이 부스**에 반응해서 말이 달라져야 한다.

## 원칙

1. **맥락 = 화면 × 행동 × 대상 × 사용자상태.** 같은 버튼이라도 어떤 부스에서 눌렀는지,
   몇 번째인지, 취향과 맞는지에 따라 말이 다르다.
2. **변주 필수.** 트리거마다 **최소 3개 변주** 풀. 연속 노출 시 같은 말 반복 금지
   (라운드로빈/무작위). 반복은 로봇 느낌.
3. **구체 정보를 문다.** 일반론 금지. 부스명·태그·굿즈·가치 등 **실제 데이터 슬롯**을
   채워 말한다. (`{booth}`, `{tag}`, `{value}`, `{n}`, `{goods}` …)
4. **결정론이 고르고, 로미는 말만.** 어떤 발화를 쓸지는 신호(반응·스크롤·검색·%)로
   서비스가 결정. 로미는 언어 표면. (CLAUDE.md 에이전트 구조와 정렬.)
5. **반말 로미 따뜻체.** 짧게, 담백하게, 판단은 사용자 몫으로 남긴다. 숫자 자랑·과장 금지.
6. **휘발성.** 대부분 하단 플로팅 발화 = 스쳐가는 말. 고정 배너로 박아두지 않는다.
7. **사용자 유형에 따라 접근 자체가 바뀐다(가장 중요).** 아래 §0 참조. 트리거 발화는
   고정 대본이 아니라 **사용자 모델로 조건화된 풀에서 선택**된다. 같은 [D-09] 끌림이라도
   고관여·고학습자와 일반·굿즈러에게 로미는 **다른 깊이·다른 어휘·다른 관점**으로 말한다.
8. **로미는 LLM 채팅처럼 계속 읽고 답한다.** 사용자가 이 서비스에서 하는 **모든 행동**
   (탭·스크롤·검색·반응·체류·재방문)이 입력이고, 로미는 거기 맞춰 반응한다 — ChatGPT가
   사용자 글에 맞춰 답하는 것과 같은 원리. 단, CLAUDE.md 속도 규칙(탭엔 LLM 금지, 즉답)과
   충돌하지 않게 **2층**으로 간다(§0-B).

# 0. 사용자 유형별 적응 (가장 중요한 축)

> "이 방식과 접근이 무수하게 계속 바뀌어야 한다." — 로미는 사용자가 하는 모든 행동을
> 읽고, **그 사람이 누구인지에 맞춰 대하는 방식 자체를** 바꾼다. 고정 대본이 아니다.

## 0-A. 적응 축 (사용자 모델 = L4)

로미의 발화·깊이·어휘·무엇을 짚는지는 아래 축들의 조합으로 결정된다. 이 축들은
관람 행동(반응·검색·체류·스크롤)으로 **실시간 갱신**된다.

| 축 | 값 | 로미가 바꾸는 것 |
|---|---|---|
| **관여도** | 고 / 중 / 저 | 말 길이·제안 밀도. 고관여=더 깊게 파고들 거리, 저관여=가볍게 핵심만 |
| **학습 성향** | 고학습 / 저학습 | 고학습=맥락·의미·과정·만든 사람 이야기 / 저학습=결과·재미·획득·비주얼 |
| **지배 가치** | 굿즈·체험·발견·학습·소통·트렌드·영감·휴식 | 무엇을 근거로 들이대는지 (굿즈러엔 굿즈·한정판, 체험러엔 해볼 거리) |
| **경험도** | 베테랑 / 초심자 | 베테랑=뻔한 것 생략·새로운 결 / 초심자=쉬운 것부터·짚어주기 |
| **현재 상태(L2)** | 시간대·피로·혼잡 | 늦었으면 짧게·붐비면 우회 제안 |

## 0-B. 2층 구조 (속도 규칙과 양립)

- **1층: 결정론 템플릿 풀(즉답).** 트리거 × 사용자유형 → 변주 풀에서 선택. LLM 없음,
  즉각. 이 카탈로그의 대부분이 여기. CLAUDE.md "탭엔 LLM 금지"를 지킨다.
- **2층: LLM 자유 대화(명시적 대화 턴).** 사용자가 로미에게 직접 말 걸거나, 깊은 설명이
  필요할 때만. 사용자 모델 + 최근 행동을 프롬프트에 넣어 진짜 대화. 로딩 UX 허용 구간.
- 즉, **평소엔 유형별 템플릿으로 빠르게 적응하고, 진짜 대화가 필요할 때만 LLM.**

## 0-C. 아키타입 예시 (조합의 축약형)

실무에선 축 조합이지만, 대표 페르소나로 감을 잡는다:

- **큐레이터형** (고관여·고학습·발견/학습·베테랑): 맥락과 의미를 원함. 로미는 배경·
  만든 사람·계보를 짚고, 뻔한 추천은 건너뛴다. 어휘도 한 단계 깊게.
- **획득형** (중관여·저학습·굿즈/트렌드): 굿즈·한정판·득템이 목적. 로미는 뭐가 있고
  언제 사야 유리한지, 놓치면 안 되는 것 위주. 짧고 실용적.
- **체험형** (중관여·체험/영감): 직접 해보는 게 좋음. 로미는 해볼 거리·체험존·인터랙션을
  앞세운다.
- **산책형** (저관여·휴식): 부담 없이 슬슬. 로미는 적게, 여유롭게, 강요 안 함.

## 0-D. 아키타입별 발화 차등 — 워크드 예시

같은 트리거 **[D-09] 끌림({booth}, {tag})** 이 유형에 따라 이렇게 달라진다:

| 아키타입 | 발화 |
|---|---|
| 큐레이터형 | "{booth}에 끌렸구나 — 여기 {tag} 쪽에서 결이 꽤 뚜렷한 곳이야. 만든 사람 이야기가 재밌어." |
| 획득형 | "오 {booth}! 여기 {goods} 물건 좋아. 굿즈 노리면 킵." |
| 체험형 | "{booth} 끌렸어? 여기 직접 해볼 게 있어 — 그거 위주로 볼래?" |
| 산책형 | "좋아, {booth} 기억해둘게. 천천히 봐도 돼." |

> 이 표 형태를 카탈로그 전 트리거로 확장하는 것이 1000+의 실체다:
> **트리거 수 × 아키타입 수 × 변주 = 발화 공간.**

# 0.5 데이터·엔진이 따라와야 한다 (DB · 브레인 · Gemini)

> "로미가 그걸 이해하려면 DB도 따라와줘야 하고 Gemini API도 마찬가지. 이게 다 어우러져야
> 한다." — 발화가 유형에 맞춰 바뀌려면, 그 유형을 **저장·갱신하는 데이터 층**과 **읽어
> 말로 바꾸는 LLM 층**이 붙어야 한다. 발화 카탈로그는 표면일 뿐.

## 0.5-A. 이미 있는 것 (현재 코드 — 재활용)

- **원장** `user_signal_log` (`0019`): append-only 행동 신호. `kind` + `booth_code` +
  `slugs`(관심 이전축). 재증류 소스.
- **종단 모델** `user_brain` (`0020`): 결정론 증류본(원장 아님, per-user 1행).
  - `literacy` {overall, byTheme, visitsCount, boothsSeenCount} → **학습성향·경험도** 축.
  - `interests[]` InterestNode {key, confidence 0..1, trend, signals:explicit/implicit/
    negative} → **지배 가치** 축 + 로미가 보여줄 **% 신뢰도**.
  - `preferences` {crowdTolerance, waitTolerance, depthVsBreadth, movement} → **관여도·
    상태** 축.
  - `visits[]` VisitDigest → L3 에피소드(회고).
- **신호 종류** `SignalKind`: booth_visited · booth_skipped · booth_bookmarked ·
  route_saved · feed_click · reaction_interested · reaction_later.
- **증류 엔진** `src/lib/memory/`(confidence·distill, 순수·결정론·테스트 완비).
- **AI 쿼리 로그** `ai_query_log`(`0014`): 검색·추천 입력 텍스트 + 추출 키워드.
- **부스 근거** `booth_enrichment`(+grounding, `0013/0021`): 발화가 물 구체 정보 소스.

## 0.5-B. 적응 축 → 브레인 필드 매핑

| §0-A 적응 축 | 백엔드 근거 | 상태 |
|---|---|---|
| 지배 가치 | `interests[].key` + confidence | ✅ 있음 |
| 학습 성향/안목 | `literacy.overall` / `byTheme` | ✅ 있음 |
| 경험도(베테랑/초심자) | `literacy.visitsCount` / `boothsSeenCount` | ✅ 있음 |
| 관여도(고/중/저) | `preferences.depthVsBreadth` (근사) | 🟡 파생 필요 |
| 현재 상태(혼잡/피로) | `preferences.crowdTolerance` + L2 실시간 | 🟡 부분 |
| 아키타입(큐레이터/획득/체험/산책) | — (위 축들의 분류) | ❌ 신규 |

## 0.5-C. 갭 — DB/신호가 따라와야 할 것

1. **신호 종류 확장** (`SignalKind` + 로깅 지점):
   - `reaction_skip`(별로, 부정) · `reaction_seen`(이미 봄) — 지금 booth_skipped 하나로
     뭉개짐. 부정/중립 분리.
   - `feed_scroll`(깊이·끝도달) — 스크롤을 신호로. 탐색 성향·완주.
   - `search_query`(검색어→관심) — `ai_query_log`를 user_signal로도 접합(D-05 "특정 부스
     검색=강한 관심").
   - `booth_dwell`(체류시간) — `dwell.ts` 있음, 신호 kind로 승격.
2. **파생 필드**(증류 결과에 추가):
   - `involvement` 0..1 (반응 밀도·체류·검색 빈도로 산출).
   - `archetype` (큐레이터/획득/체험/산책 — 축 조합 분류, 소프트 라벨 + 확신도).
   - `tasteProgressPct` (온보딩 진행률 = interests 상위 confidence 집계 → 로미가 %로 발화).
3. **마이그레이션 필요**: 위 kind 추가는 스키마 변경 아님(text). archetype/involvement/
   progress는 `user_brain.data` jsonb라 **컬럼 변경 없이** distill 확장으로 가능.

## 0.5-D. Gemini 층 (2층 LLM — §0-B와 짝)

- **현재 Gemini 사용처**: 온보딩 추론(`onboarding-inference`) · 부스 추천 리랭크
  (`booth-recommender`, grounding) · 회고 서술(`recap`).
- **적응을 위해 붙일 것**: LLM 호출(2층)의 프롬프트에 **브레인 요약(아키타입·안목·상위
  관심·involvement) + 최근 신호 몇 개**를 주입 → 같은 질문도 유형별로 다른 결의 답.
  - 예: 큐레이터형이면 "맥락·계보 위주로, 한 단계 깊은 어휘로" 시스템 지시.
  - 예: 획득형이면 "굿즈·한정·타이밍 위주로, 짧고 실용적으로".
- **속도 규칙 준수**: 1층 템플릿(즉답)이 평소를 담당, LLM은 명시적 대화/깊은 설명에만.
  `thinkingBudget=0` 유지(CLAUDE.md).
- **RAG**: 부스별 구체 발화(피드 차별화)는 `booth_enrichment`(요약·굿즈·recommendation
  Reasons·valueTags)를 소진 — LLM 없이도 결정론으로 부스마다 다른 근거 조립 가능.

| 슬롯 | 뜻 | 예 |

## 발화 스키마 (각 항목 형식)

```
### [TRIGGER-ID] 트리거 설명
- 신호: 무엇을 감지했나 (결정론 조건)
- 대상: 어느 화면/컴포넌트에 뜨나
- 변주: (아래는 유형 무관 기본 풀. 유형 차등이 필요하면 §0-D처럼 아키타입 표로.)
  1. "..."
  2. "..."
  3. "..."
- 유형 차등(선택): 큐레이터형 / 획득형 / 체험형 / 산책형 별 발화가 다르면 명시
- 기억: 이 발화가 반영/저장하는 신호 (L2/L3/L4)
```

> A~J 시드는 대부분 **유형 무관 기본 풀**로 먼저 깔았다. 확장 = 각 트리거에 §0-D 같은
> 아키타입 차등을 채우는 것.

# 커버리지 체커 (1000+ 향해)

**1000+의 산식**: 발화 공간 = **트리거 수 × 아키타입 수(≈4) × 변주(≈3)**. 아래는
*트리거* 기준 목표. 여기에 아키타입 차등을 곱하면 실제 발화가 1000+로 벌어진다.

각 섹션을 트리거 단위로 계속 쪼갠다. 아직 얕은 곳 = 확장 대상.

- [ ] A 앱 진입/언어/소개 — seed 2 / 목표 ~15
- [ ] B 앱 온보딩 문답 — seed 5 / 목표 ~80 (질문×답변 조합)
- [ ] C 전시 목록/추천 — seed 3 / 목표 ~30
- [ ] D 전시 온보딩(검색·피드·반응) — seed 18 / **목표 ~300** (반응×태그×누적×%)
- [ ] E 전시 홈 — seed 3 / 목표 ~40
- [ ] F 지도 — seed 4 / 목표 ~60
- [ ] G 부스 상세 — seed 4 / 목표 ~120 (섹션×부스유형×굿즈유무)
- [ ] H 메모/커뮤니티/회고 — seed 3 / 목표 ~80
- [ ] I 로그인/계정 — seed 2 / 목표 ~20
- [ ] J 시스템(로딩/에러/빈) — seed 3 / 목표 ~40
- [ ] K 엣지: 오프라인·권한거부·업로드실패·세션만료·연속 별로·되돌리기 — 미착수 / 목표 ~100
- [ ] L 시간대/혼잡/날씨 등 상황(L2) 결합 — 미착수 / 목표 ~100

> **확장 방식 제안**: 섹션 하나씩 트리거를 전수 분해해 변주까지 채운다. 대량 병렬
> 저작이 필요하면 워크플로우로 팬아웃(사용자 승인 시).
